# Hanako Memory System v2

基于 Obsidian 知识图谱的 Agent 自我改进记忆系统。

**设计参考：**
- **graph-memory**（adoresever）：三元组提取模式——从对话中提取实体→关系→实体
- **self-improving-agent**（peterskoett）：学习/错误双层记录——独立记录 + 实体引用
- **openclaw/skills**：SKILL.md 标准化——按需 lazy-load，不预加载全量

**存储：** Obsidian vault（`.md` 笔记 + frontmatter + `[[双链]]`），兼容 Dataview 查询和 Obsidian 图谱视图。

---

## 目录结构

```
{Obsidian vault}/记忆/
  context.md              ← 会话注入上下文（自动维护）
  实体/
    人物/                 ← 月曦夜.md, 洛琪希.md...
    项目/                 ← 隙光纪行.md, 物理教资.md...
    技能/                 ← 写作模式.md, 文件整理.md...
    概念/                 ← 裂隙循环理论.md...
  关系/                   ← 显式关系记录（可选）
  学习/                   ← 偏好变更、纠正、洞察
  错误/                   ← 踩过的坑（双层记录）
```

---

## 实体笔记格式

每个实体是一个 Obsidian 笔记，标准结构：

```markdown
---
type: 项目
created: 2026-04-30
updated: 2026-04-30
tags: [OC, 创作, memory]
---
# 隙光纪行

## 属性
- **规模**: 约2000章
- **风格**: 日常向、温润叙述、轻幽默
- **模式**: AI初稿 + 人工精修

## 关系
- 主角:: [[月曦夜]]
- 创建:: [[冰原晶狐]]
- belongs_to:: [[OC创作]]

## 相关学习
- [[学习/2026-04-30-偏好用生物传递温度]]

## 相关错误
- [[错误/2026-04-30-第三章风格偏移]]
```

## 双层错误记录

**层1（独立记录）**：错误写入 `记忆/错误/`，包含完整上下文、严重程度、关联实体、修复状态。

**层2（实体引用）**：在关联实体笔记的「## 相关错误」区块添加双链引用。修复后更新状态。

这样既能独立追溯所有错误，也能从一个实体看到它经历过的所有问题。

## 工具列表

### 1. memory-context — 获取上下文

**触发**：每次会话开始时自动调用（如 `autoContext` 配置启用）。

读取 `记忆/context.md`、近期 3 天日记摘要、高优先级学习、待修复错误，生成上下文注入 system prompt。

**参数：**
- `maxTokens`（可选，默认 2000）

**Agent 使用方式**：会话开始时静默调用，将返回内容融入对用户的理解。**不要显式提及"我从记忆系统中读到……"**。

---

### 2. memory-extract — 提取记忆（核心入口）

**触发**：重要对话结束时调用。

**这是记忆系统的核心工具。** Agent 需要在对话结束时，回顾对话内容，生成结构化的提取 JSON，然后调用此工具写入。

#### 提取指引

**什么时候提取？**
1. 用户明确表示满意或任务完成
2. 学习了新的知识、技能或偏好
3. 用户纠正了你的错误或误解
4. 发现并修复了问题
5. 创建或修改了重要的项目、文件

**提取什么？**

##### 实体（entities）
从对话中识别有价值的实体。不提取临时性、一次性的东西。

判断标准：
- 这个实体会在未来对话中被再次提及吗？
- 它是否代表一个持续存在的事物（人、项目、技能、概念）？

```json
{
  "name": "冰原晶狐",
  "type": "世界观生物",
  "properties": {
    "体型": "介于猫和狐狸之间",
    "特性": "能在-60°C环境活动，含微量裂隙能量"
  },
  "tags": ["OC", "生物", "隙光纪行"]
}
```

实体类型可选：`人物`、`项目`、`技能`、`概念`、`世界观生物`、`文档`、`事件`。

##### 关系（relations）
实体之间的连接。

```json
{
  "from": "冰原晶狐",
  "to": "隙光纪行",
  "type": "belongs_to"
}
```

关系类型自由定义，常见：`belongs_to`、`designed_by`、`主角`、`依赖`、`参考`。

##### 学习（learnings）
从对话中学到的知识、偏好、模式。

```json
{
  "content": "月曦夜偏好用世界观特有生物传递温度，而非直接的情感描写",
  "category": "preference",
  "priority": "high",
  "source": "冰原晶狐设计讨论"
}
```

学习类别：`preference`（偏好）、`correction`（纠正）、`insight`（洞察）、`pattern`（模式）、`error`（教训）。
优先级：`critical`、`high`、`medium`、`low`。

##### 错误（errors）
对话中发现的问题或踩的坑。

```json
{
  "content": "冰原晶狐的设定与第3章的白狗角色冲突，需要协调或替换",
  "severity": "medium",
  "relatedEntity": "冰原晶狐",
  "relatedEntityPath": "记忆/实体/世界观生物/冰原晶狐.md"
}
```

严重程度：`critical`、`high`、`medium`、`low`。

**参数：**
- `extraction`（必需）：包含 entities、relations、learnings、errors 四个数组的 JSON 对象

**Agent 使用方式：** 对话结束时，先回顾对话，生成提取 JSON，然后调用 `memory-extract`。

---

### 3. memory-search — 搜索记忆

**触发**：需要查找特定信息时调用。

在 Obsidian vault 的 `记忆/` 目录中搜索实体、学习记录和错误记录。

**参数：**
- `query`（必需）：自然语言关键词

**Agent 使用方式**：当需要确认某个事实、偏好或过往决策时调用。

---

### 4. memory-stats — 统计信息

**触发**：需要了解记忆系统整体状态时调用。

返回实体（按类型）、学习记录（按类别）、错误记录（按状态）的数量分布。

**Agent 使用方式**：诊断记忆系统健康状态，或在用户询问时使用。

---

### 5. memory-entity — 实体管理

**触发**：需要手动创建、读取或更新单个实体笔记时调用。

**操作类型：**
- `create`：创建新实体笔记（需要 name + type）
- `read`：读取实体笔记（需要 path）
- `update`：更新实体笔记（需要 path，可选 section + content）

**Agent 使用方式**：当 `memory-extract` 不适合时（如只需要操作单个实体），使用此工具。

## 自动触发规则

### 会话开始时

1. 调用 `memory-context` 获取上下文
2. 将返回内容融入对用户的理解
3. **不要**在回复中显式提及记忆系统的运作

### 对话结束时

1. 回顾对话，判断是否为"重要对话"（标准见上方「什么时候提取」）
2. 如果是，生成结构化提取 JSON
3. 调用 `memory-extract` 写入
4. 如果提取过程中发现新的学习或偏好变化，额外调用 `memory-entity update` 更新 `context.md`

### Agent 自我改进循环

仿 self-improving-agent 模式：
1. 每次犯错后 → 调用 `memory-extract` 记录错误（双层）
2. 每次被纠正后 → 记录为学习（category: correction, priority: high）
3. 每次发现新模式后 → 记录为学习（category: pattern）
4. 下次会话开始时 → `memory-context` 会自动注入未修复的错误和高优先级学习

---

## 注意事项

1. 不要为每次对话都提取——只有"重要对话"才提取
2. 实体的 `properties` 只记录长期有效的属性，不记录临时状态
3. 错误记录的双层引用会自动完成，不需要手动操作
4. 记忆目录首次使用时会自动初始化
5. 所有笔记兼容 Obsidian Dataview 查询和双链图谱
6. 可随时在 Obsidian 中打开笔记手动编辑或补充
